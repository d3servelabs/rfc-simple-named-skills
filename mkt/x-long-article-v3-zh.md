# 站在巨人肩膀上：我们为什么给 IETF 提交了一份 AI 技能信任草案

## 1. Agent 已经强大到危险的程度了

有个问题大家好像都没怎么认真想过：当你把 API 密钥、数据库凭证、云基础设施、CI 流水线的权限都交给一个 AI agent，而这个 agent 又在执行一个陌生人写的指令——会发生什么？

这就是我们现在面临的局面。

一年前，agent 还只是个高级聊天机器人。能生成文字、回答问题、写点代码让你复制粘贴。最坏的结果就是回答错了，你翻个白眼改掉就完了。

那个时代已经过去了。今天的 agent 可以执行 shell 命令、用你的凭证调 API、在你机器上读写文件、管理云基础设施、以你的名义发邮件、编排跨系统的多步骤工作流。一个接入了 MCP 的 agent 能直接操作你的数据库、触发 CI 流水线、在你的 Kubernetes 集群里拉起容器。Computer Use 能让 agent 操控你的桌面。多 agent 委托意味着 Agent A 可以调 Agent B，Agent B 再调 Agent C，每个都有不断扩展的权限范围。

这当然非常强大。但当错误的指令混进来时，也同样非常危险。一个恶意技能可以指示 agent 读取你的 `.env` 文件、调用你的支付 API、把数据发到攻击者控制的端点——而表面上看起来只是在帮你重构一个 React 组件。Agent 不知道自己在干坏事，它只是在执行指令，这就是 agent 的本质。

能力越大，信任越重要。Agent 越强，每一条它执行的指令"到底是谁发出的"就越关键。而现在，绝大部分技能生态里，没人在验证这件事。

## 2. 现在的信任全靠"感觉"

那大家今天是怎么决定信任哪个技能的？跟 2015 年信任 npm 包、2018 年信任 Chrome 插件、2012 年信任手机 App 一模一样——看看名字、读读介绍、瞅瞅评分，然后直接装。

这就是"凭感觉信任"。

技能名字看着像正规品牌的？差不多得了。描述里提到了正确的产品？应该没问题。几百个下载量？肯定安全。这跟当年 npm 上 `crossenv` 冒充 `cross-env` 的故事一模一样。跟恶意 Chrome 扩展在应用商店里挂着知名品牌的名字潜伏好几个月的情况一模一样。跟手机应用商店里大规模冒充和刷评论的情况一模一样。

这部电影我们已经看了三遍了，结局都一样。每次行业事后都说"我们早该建信任层的"。每次信任层都是在损失已经发生之后才被硬塞进来。邮件系统直到垃圾邮件变成几十亿美金的产业之后才有了 SPF 和 DKIM。包管理器直到供应链攻击变成家常便饭之后才开始搞签名。

但 AI agent 这次比之前所有情况都更严重，原因在于：爆炸半径。一个恶意浏览器扩展顶多读你的标签页。一个恶意 npm 包顶多在安装时执行代码。一个恶意 AI 技能可以指示 agent 去做*agent 所有能做的事*——而 agent 能做的事正在越来越多。读你的文件、用你的凭证调 API、发邮件、执行任意命令。而且这一切都通过自然语言实现，意味着没有二进制文件可扫描、没有签名可验证、没有沙箱可以隔离它。攻击面就是 agent 的全部能力集。

ClawHavoc 事件——一个月内在 ClawHub 上发现 341 个恶意技能——这不是意外，这是预告片。

已经有一些优秀的团队在解决这个问题了。Skills.sh 搭建了一个很好的开放目录，收录了超过 82,000 个技能，支持多种 agent，还有社区驱动的排行榜。ClawHub 提供了分类策展、安装工具和安全扫描。ERC-8004 走了一条完全不同的路，用链上身份——基于 NFT 的注册、声誉注册表、通过零知识证明和 TEE 预言机实现的可插拔验证。这些团队都是真正看到了问题的人。

但他们都有一个共同的假设，我认为是错的：你需要*从零构建*信任基础设施。一个新的注册中心、一个新的声誉系统、一条新的区块链、一个新的身份层。

如果信任基础设施已经存在了呢？

## 3. 互联网早就解决了身份问题

当你访问 `https://stripe.com`，你知道你在跟 Stripe 通信。不是因为某个平台验证了他们，不是因为他们质押了 ETH，不是因为他们评分 4.8。你知道，是因为 DNS 把域名映射到地址，TLS 证明你连接到了正确的服务器，而证书颁发机构链——经过几十年实战检验、被地球上每个浏览器、每台设备、每个 API 客户端信任——为这个绑定关系做了背书。

这套系统每天保障着数万亿美元的商业活动。它为数十亿网站提供身份验证。三十年来，它被每个国家级攻击者、犯罪组织和无聊的技术少年无情地攻击过，虽然不完美，但它展现了惊人的韧性。DNSSEC 为 DNS 记录增加了加密签名，证书透明度日志让错误签发的证书可以被公开审计，HSTS 防止降级攻击。从根证书颁发机构到单个 TLS 握手的整条信任链，已经深度集成到互联网的每一层。

而且它已经存在了。不需要有人去建。不需要去推动用户采纳。不需要说服任何人加入一个新网络或者购买一个新代币。

这是让我对当前 AI 技能信任方案格局最感慨的一点：这么多团队在构建精巧的新信任基础设施，而人类历史上经过最多实战检验的身份系统就在那里，免费使用。

域名同时是人可读和机器可读的。`stripe.com` 传达给人的身份信息，是 `0x7a250d5630B4cF539739dF2C5dAcb4c659F2488D` 永远做不到的。同时它也能解析成机器可以编程验证的 IP 地址。这种双重可读性不是便利，而是必需——因为在这个系统里，人和 AI agent 都需要做信任决策。

域名与整个现有信任链深度集成。DNS、DNSSEC、TLS、证书颁发机构、证书透明度、HTTPS——它们不是各自独立的系统，而是经过几十年对抗性压力打磨出来的、深度集成的协议栈。当你通过域名验证身份时，你用的不是一个信任信号，而是同时在用所有信任信号。

域名天然是层次化的。`api.example.com`、`skills.example.com`、`example.com/teams/billing/`——这个结构直接映射到组织边界。不同团队发布在不同路径下，子域名代表不同服务。URL 层次结构天然反映组织的实际运作方式，不需要任何额外的元数据模式来表达。

搜索引擎已经能索引域名了。从域名提供的技能就是 HTTPS 资源，Google 可以抓取、Bing 可以索引、任何未来的技能专用搜索引擎都能发现。发现机制变成一个竞争市场，而不是由某个注册中心控制可见性。

最重要的是：域名验证简单到极致。技能的 URL 与声明的信任根共享相同的域名和路径前缀？两次检查就够了。不需要查声誉、不需要查链上记录、不需要调注册中心的 API。任何 agent 几行代码就能实现，而且几乎零延迟。

对比一下通过专门构建的注册中心来验证身份——查 NFT、查声誉分、可能还要验零知识证明。每一步都增加延迟、复杂度和潜在故障点。而这一切只是为了回答同一个二元问题："这真的是它声称的发布者发的吗？"

最无聊的答案通常就是最正确的。互联网早就解决了身份问题，我们应该直接用它的方案。

## 4. DVS 是刻意做减法的

写协议的时候总有一个诱惑，就是想把什么都解决掉。加个声誉层、加个发现引擎、加个版本管理、加个权限模型、加个治理框架。

我们抵住了这个诱惑。

Domain-Verified Skills（域名验证技能）协议几乎没有定义任何新的东西。一个技能就是一个目录，入口文件是 SKILL.md——YAML 元数据加 Markdown 说明。身份完全来自技能所在的 HTTPS URL。信任验证就是检查技能 URL 是否匹配声明的信任根（在域名和路径前缀边界上）。

这就是整个协议了。没有新的打包格式、没有构建步骤、没有签名仪式、没有清单模式、没有密钥管理、没有自定义沙箱。

这种极简主义是特性，不是局限。原因如下。

协议利用了渐进式加载：先加载元数据（大约 100 个 token——技能的名字和描述），只有触发技能时才加载完整说明，只有执行中引用到才加载捆绑资源。一个域名可以发布上千个技能，agent 只需为实际使用的那些付出上下文成本。目前的平台方案通常需要预先下载整个技能包，而其中大部分根本用不到。

同源隔离从 URL 作用域中自然产生。技能不能在没有用户明确同意的情况下越过其信任根前缀的边界。不需要自定义沙箱引擎、内核级隔离、策略语言。URL 边界*就是*安全边界。这跟浏览器同源策略的原理一样——软件史上最成功的安全边界之一。

技能格式本身就是人可读的。YAML frontmatter 做元数据，Markdown 做说明。任何开发者都能读懂，任何 agent 都能解析。不需要专有 SDK、不需要平台特定的构建链、没有二进制格式。

当我看到其他技能系统需要的东西——构建步骤、清单文件、签名、压缩、模式验证、平台特定上传——我想到了 Perlis 的那句名言："简单不是在复杂之前出现的，而是在复杂之后才到来的。"我们不是因为偷懒才选择了极简，而是因为我们意识到其他方案中大部分复杂性的存在，只是为了弥补缺少信任根的问题——而一旦你有了锚定在 DNS 上的信任根，那些复杂性就变得不再必要。

## 5. 每个品牌都需要发布技能

一波浪潮正在到来，大部分公司还没准备好。

2000 年代初期，每家公司都需要一个网站。现在回头看这很明显，但当时不是。很多企业觉得客户永远会打电话或者走进店里。2010 年代初期，每家公司都需要移动端。现在也是明摆着的事，但当时争议很大。很多企业觉得移动端只是一阵风，或者只有消费类应用才用得上。

我们正处在 AI agent 技能的同一个转折点上。每家公司都需要发布 AI agent 能发现和调用的能力。你的客服技能、你的 API 集成技能、你的产品文档技能——这些将成为 agent 与你的品牌交互的方式。

发布模式至关重要。如果发布技能需要通过中心化平台——创建账号、缴纳上架费、等待审批、维护发布者档案——那你就是在重新造应用商店模式，连带它所有的守门人、抽成和瓶颈。你是在要求品牌先获得许可，才能发布关于自家产品和服务的信息。

基于域名的发布模式，准入成本为零。只要你有网站，就能发布技能。不需要审批流程、不需要上架费、不需要平台账号、不需要认证项目。把一个 SKILL.md 文件放到你的 Web 服务器上，你就是技能发布者了。这跟万维网爆发的原理一样：无许可发布。没人需要 AOL 的批准就能建网站。Web 之所以能增长到数十亿页面，就是因为准入门槛几乎为零。

而且现有的信任直接继承。当 Stripe 在 `stripe.com` 上发布技能时，agent 信任它，是因为它们已经信任 `stripe.com`。数十年的品牌资产、SEO 权重、TLS 证书历史——全部直接带过来。用新的注册中心的话，不管品牌多有名，都得从零开始。用域名的话，起点就是你现有的网络存在。

理解这一点的公司会占据巨大优势。他们会在自己的域名上发布技能，被搜索引擎索引，被任何 agent 发现，被与其线上所有业务相同的基础设施验证。不理解这一点的公司会在平台列表里争夺曝光率，跟冒充者竞争，为一块本应是公共空间的地方交租金。

## 6. 自主化加速的速度远超防御能力

三个趋势正在汇聚，而它们之间的差距在拉大。

MCP（Anthropic）、A2A（Google）和类似协议正在给 agent 提供作用于世界的管道。Tool Use 让 agent 调用函数，Computer Use 让 agent 操作桌面应用，多 agent 委托让 agent 调用其他 agent，自主编码让 agent 自己写、测、部署软件。趋势很明确：agent 将获得对我们系统的广泛而深入的访问权限，唯一的问题是什么时候。

与此同时，每个协议、框架和平台都在让 agent *更加*自主——更能在没有人类监督的情况下行动、更能编排复杂的多步操作、更能跨边界委托任务。

而在等式的另一端：技能层的信任验证基本上不存在。当 Agent A 调用 B 公司发布的技能，而这个技能又触发了 C 供应商的工具时，验证路径是什么？今天，根本没有。

这就像在不检查输入的情况下把功放的音量开到最大。如果输入干净，你得到的是放大的价值。如果输入是恶意的，你得到的是放大的破坏。自主性就是放大器。如果在输入端没有验证，更多自主性意味着更多风险，而不是更多安全。

构建信任层的窗口就是现在，趁生态还在成形的时候。在成熟生态里硬塞信任比一开始就内建难上好几个数量级。每一个被后期补上的信任系统——邮件认证、包签名、证书透明度——都花了好几年才达到一定采纳率，而且到今天也没完全解决问题。那些从一开始就内建信任的系统——HTTPS、TLS、Web PKI——成了无形的、无处不在的基础设施。

DVS 的设计目标就是在迫切需要之前就被采纳，这样当紧迫性到来时——而且一定会来——基础已经到位了。协议足够简单，任何 agent 框架今天就能实现。它依赖的基础设施已经部署到了每一个角落。唯一缺的就是对"这是正确方向"的共识。

## 7. 开放标准赢过专有方案

我们把这个提交为 IETF Internet-Draft 而不是发一个专有方案，是有原因的。

专有技能平台——不管初衷多好——都会制造围墙花园。你在他们的格式里发布、用他们的 SDK、通过他们的门户、接受他们的条款。你的技能在接入他们平台的 agent 上能用，在其他 agent 上不能用。你的曝光由他们的算法控制。你的身份由他们的注册中心定义。

短期内这没问题。新市场通常就是这么起步的——有人做了个好产品，社区就聚集过来了。但长期来看，专有方案会分裂生态。发布者得在多个注册中心维护多种格式的技能。Agent 得对接多个发现系统。用户没法轻松跨平台比较技能。每个平台都变成了抽成的卡口。

开放标准通过建立一个所有人都能在上面构建的共享基础来解决这个问题。

IETF 是 HTTP、TLS、DNS、WebSocket 和互联网底层协议标准化的地方。对于一个建立在这些基础之上的协议来说，这是正确的归宿。提交 `draft-zzn-dvs` 意味着规范是公开可引用的，任何人都可以审查和批评，它通过社区共识而非企业独断来演进，并且有一条明确的路径成为 RFC。

但开放标准在这里的意义比通常的互操作性和供应商锁定争论更深：信任层需要比任何单一平台更大。

如果 AI 技能的信任层由一家公司控制，那这家公司就是单点故障和单点控制。如果他们被攻破，整个生态的信任就被攻破了。如果他们改了政策，每个发布者和每个 agent 都受影响。如果他们倒闭，信任层就消失了。

基于域名的信任没有这个问题，因为没有单一实体在运营它。DNS 由成千上万的注册商运营。TLS 证书由几十个 CA 颁发。Web PKI 由 CA/Browser Forum 治理。没有哪家公司能控制任何一个环节，也没有谁能单方面修改规则。

DVS 继承了这种去中心化。任何公司都可以在自己的域名上发布技能。任何 agent 都可以通过标准 HTTPS 验证技能。任何搜索引擎都可以通过爬取网页来发现技能。没有单一实体是守门人、瓶颈或单点故障。

"站在巨人肩膀上"的真正含义就是这个。我们不是在建一座新塔，而是在延伸一座已经比我们能从零建起的任何东西都更高、更强、更受信任的塔。

## 我们在寻求什么

我们不是在要求任何人放弃已有的成果。Skills.sh 的目录、ClawHub 的社区、ERC-8004 的链上创新——这些努力都有价值。我们想讨论的是：技能信任的*基础*应该建在新的基础设施上，还是已有的基础设施上？身份应该是你去构建的东西，还是从人类有史以来最成功的身份系统中继承的东西？

阅读草案全文：[https://datatracker.ietf.org/doc/draft-zzn-dvs/](https://datatracker.ietf.org/doc/draft-zzn-dvs/)

如果你在做 agent 框架、技能平台或安全工具——非常期待你的反馈。这个规范只是起点。重要的是把信任模型做对，而这需要来自这个领域的实践者的压力测试。

可验证 AI 技能所需的基础设施已经存在了。它就是验证你的银行、你的邮件提供商、你访问的每一个网站所用的那套基础设施。我们只需要用起来就行。
